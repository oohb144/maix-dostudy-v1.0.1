- 发现：`main.py::_on_fusion_play()` 在 UI 回调中同步调用 `self._video_player.play(av_path)`；如果播放器内部没有强制刷新显示或上下文识别不准，终端会打印播放而屏幕仍停留在原 UI。
- 发现：`main.py::_handle_recording()` 进入识别后录制时，每轮仍执行 `detect_and_recognize(img)`，随后还要做 `img.to_format(FMT_YVU420SP)` + `video.Encoder.encode()`，这会同时占用 NPU 和视频编码，卡顿是预期结果。
- 发现：`main.py::_on_enter_recording()` 与 `_on_enter_manual_recording()` 都调用同一个 `RecorderManager.start_recording()`；当前识别录制和纯录制都会启动音频线程，不符合“识别时只录视频”的新要求。
- 发现：`main.py::_sync_stream_settings()` 在主循环里反复检查并启停 HTTP/RTSP 服务；若浏览器/VLC 正持有连接，关闭时会同时释放 server、camera、audio_recorder，容易与下一帧 `write()` 或显示链路竞争，造成卡顿或闪退。
- 修复方向：播放器改成更接近官方示例的稳态播放路径，并增加帧显示日志；录制器支持“是否录音”参数，识别录制禁音、纯录制保留音频；识别录制进一步降频/复用缓存；推流关闭增加保护状态，先停写再停服务，避免 stop 与 write 并发冲突。

## 2026-06-11 用户澄清 - 仅手动录制模式需要优化

- 用户明确说明：“录制模式还是卡” 指的是点击触摸“录制”按钮进入的手动纯录制模式，不是识别模式或识别后自动录制模式。
- 用户要求：识别链路恢复原来的行为，不接受对 `RECOGNIZE_DETECT_INTERVAL_MS` / `RECORD_RECOGNIZE_INTERVAL_MS` 的识别降频改动。
- 保留改动：识别状态下录制仍然只保留视频，不录音；手动纯录制模式继续保留音视频录制。
- 后续优化方向应聚焦手动纯录制模式自身的编码/显示/推流竞争，而不是动识别路径。

## 2026-06-11 ffmpeg 参数兼容性回归

- 用户日志证据：板载 ffmpeg 在执行 `-c:v libx264 -preset veryfast ...` 时直接报 `Unrecognized option 'preset'`，返回码 2048，导致融合 0/3 全失败。
- 判断：当前设备内置 ffmpeg 为裁剪版本，虽然带有 `libx264`，但不支持完整的 x264/preset 参数集合，不能直接套用桌面版命令。
- 处理：融合命令改为两级回退：先尝试不带 `preset` 的精简 H264/AAC 命令；若仍失败，自动退回此前可工作的 `-c:v copy -c:a aac -shortest` 方案，优先恢复“至少能融合成功”。

## 2026-06-11 融合视频播放有画面但无声音

- 用户反馈：当前融合视频已经可以在屏幕上播放画面，但没有声音输出。
- 判断：`video_player_manager.py` 主播放路径已经切到 `video.Decoder.decode_video()`，该路径只返回视频帧，不会带出音频上下文，所以不会触发原来的 PCM 播放分支。
- 判断：播放对象通常是 `_av.mp4`，若直接按文件名回找同名 WAV，会错误地查找 `*_av.wav`；正确来源应回退到原始基名对应的 `*.wav`。
- 处理：播放器改为“视频画面用 `decode_video()`，声音独立用 `audio.Player(wav_path)` 线程播放”的稳妥方案；并在查找音频时兼容 `_av.mp4 -> 原始 .wav` 的映射。

## 2026-06-11 音视频同时播放可行性分析

- 文档证据：`video/play.md` 官方示例只展示 `video.Decoder.decode_video()` + `ctx.image()` + `disp.show(img)` 的纯视频播放路径，没有给出 MP4 内嵌音频同步播放示例。
- 文档证据：`audio/play.md` 明确支持 `audio.Player("/root/output.wav").play()` 播放整段 WAV，也支持 `audio.Player(sample_rate=..., channel=..., block=False)` + `play(pcm)` + `get_remaining_frames()` 的流式 PCM 播放。
- 可行性判断 1：如果目标只是“画面和声音大致同时开始”，这是可做的。做法是视频线程按 `ctx.duration_us()` 播放画面，音频侧改成非阻塞 `audio.Player(block=False)` 或单独线程，在视频第一帧前后立即启动音频。
- 可行性判断 2：如果目标是“严格同步、长时间不漂移、接近播放器级 AV Sync”，当前 MaixPy 文档没有给出成熟现成方案，难度明显更高，存在音频时钟和视频时钟漂移问题。
- 风险点：当前独立 `wav.play()` 很可能在调用阶段会先把整段数据灌入内部缓冲，因此实际听感可能表现为“声音先起，画面后起”；即便改成并行启动，也未必天然严格对齐。
- 更优方向：优先尝试 `Decoder.decode()` 统一解码音视频帧，再用 `ctx.media_type()` 分流，视频按 `duration_us()` 显示，音频用非阻塞 PCM 队列持续写入 `audio.Player`。这条路线理论上最接近真正同步，但要看当前固件是否稳定输出音频上下文。
- 结论：能做到“基本同时播放”，大概率可以；能做到“稳定且精确同步”，目前只能说有希望，但不敢在没上板验证前承诺。

## 2026-06-11 音视频近同步播放实现

- 问题根因：此前 `audio.Player(wav_path).play()` 走的是整段 WAV 文件播放路径，容易在起播阶段先快速写入大量音频缓冲，导致用户感知为“音频明显先于视频”。
- 处理：播放器新增 `WavStreamPlayer`，按 WAV 头解析采样率/声道后，用 `audio.Player(sample_rate=..., channel=..., block=False)` 以分块方式持续写入 PCM。
- 处理：在 `decode_video()` 路径中，将音频流启动时机放到首帧视频附近，而不是提前整段启动，目标是实现“画面和声音差不多一起出来”。
- 约束：当前实现优先解决明显先后顺序问题，不承诺严格 AV Sync；长视频仍可能存在轻微漂移，需要后续上板再观察。
